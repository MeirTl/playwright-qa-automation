# Заметки по автоматизации (Technical Design)

## Архитектурный паттерн
Выбран **Page Object Model (POM)**.
*   **Почему?** Позволяет отделить логику тестов от структуры HTML-страниц. При изменении UI достаточно обновить локатор в одном классе `Page`, а не переписывать десятки тестов.

## Выбор локаторов
Приоритет отдается:
1.  `page.getByRole()` — для доступности и имитации поведения пользователя.
2.  `id` и `data-testid` — для стабильности (если добавлены в приложение).
3.  Текстовые селекторы — для проверки видимого контента.

## Обработка асинхронности
Playwright имеет мощный механизм **Auto-waiting**. В отличие от Selenium, здесь не нужно вручную проставлять `sleep()`. Тесты ждут появления элемента, его кликабельности и завершения анимаций автоматически.

## Тестирование API (Mocking)
В проекте реализован пример использования `page.route()`.
*   **Зачем?** Это позволяет тестировать фронтенд в изоляции от бэкенда. Мы можем имитировать ошибки сервера (500), задержки или специфические наборы данных, не меняя реальную БД.

## CI/CD Интеграция
Конфигурация `.github/workflows/playwright.yml`:
*   Запускает тесты на каждой ветке.
*   Сохраняет артефакты (отчеты) на 30 дней.
*   Использует параллелизацию для ускорения прогона.
